Division de tareas para 2 personas
==================================

Objetivo
--------
Dividir el trabajo de ScopeShield para que ambas personas puedan usar IBM Bob IDE de forma clara, sin bloquearse ni pisar el trabajo del otro.

Division recomendada
--------------------

Persona A: Producto + Frontend
Responsabilidad principal: que la app se vea bien, sea entendible y tenga una demo clara.

Tareas:
- Disenar el flujo principal de la app.
- Crear pantalla para pegar una solicitud ambigua.
- Crear pantalla para cargar, seleccionar o simular un repositorio.
- Disenar el dashboard del Scope Contract.
- Mostrar visualmente:
  - Scope Risk Score.
  - Resumen del pedido.
  - Riesgos tecnicos.
  - Preguntas faltantes.
  - Respuesta lista para cliente.
  - Checklist de implementacion.
- Preparar la historia de demo.
- Preparar el pitch y el guion de presentacion.

Prompts utiles para Persona A en IBM Bob:
- Build a clean React dashboard for this Scope Contract JSON.
- Improve this UX for a hackathon demo.
- Create responsive components for risk, questions, estimate and client reply.
- Make the interface feel like a serious developer tool, not a generic AI chatbot.

Persona B: Backend + Bob Logic
Responsabilidad principal: que el analisis sea util, estructurado y creible.

Tareas:
- Definir la estructura del reporte JSON.
- Crear prompts para IBM Bob.
- Analizar estructura del repositorio.
- Detectar archivos, modulos o areas relevantes.
- Generar:
  - Impacto tecnico.
  - Riesgos.
  - Preguntas de aclaracion.
  - Estimacion.
  - Plan de implementacion.
- Crear ejemplos de prueba con solicitudes ambiguas.
- Exportar o documentar el uso de IBM Bob.

Prompts utiles para Persona B en IBM Bob:
- Analyze this repo structure and map this vague client request to likely impacted files.
- Generate a structured Scope Contract JSON.
- Identify hidden scope and implementation risks.
- Create test cases for ambiguous software change requests.
- Find which files would likely be affected by this change request.

Flujo de trabajo sugerido
-------------------------
1. Ambos acuerdan el formato del Scope Contract.
2. Persona B crea una respuesta mock primero.
3. Persona A construye la UI usando ese mock.
4. Persona B conecta la logica real o semirreal.
5. Ambos prueban con 2 o 3 casos demo.
6. Persona A prepara pitch, demo visual y narrativa.
7. Persona B prepara explicacion tecnica y evidencia de uso de IBM Bob.

Contrato entre ambos: Scope Contract JSON
-----------------------------------------
Este JSON permite que frontend y backend trabajen en paralelo.

Ejemplo:

{
  "requestSummary": "",
  "hiddenScope": [],
  "impactedAreas": [],
  "riskScore": 0,
  "risks": [],
  "clarifyingQuestions": [],
  "estimate": {
    "complexity": "",
    "timeRange": ""
  },
  "implementationPlan": [],
  "clientReply": "",
  "checklist": []
}

Como documentar el uso de IBM Bob
---------------------------------
Cada persona debe guardar evidencia de como uso Bob.

Formato recomendado:
- Que se le pidio a Bob.
- Que archivo, modulo o idea genero.
- Que se edito manualmente.
- Que resultado final se integro al proyecto.

Ejemplo:
- Prompt usado: "Build a clean React dashboard for this Scope Contract JSON."
- Resultado: Bob genero una primera version del dashboard.
- Edicion manual: se ajustaron textos, estilos y estructura.
- Integracion: se agrego como pantalla principal de ScopeShield.

Recomendacion final
-------------------
Para competir bien, ambos deben usar Bob en partes visibles del proyecto.

Una buena division seria:
- Persona A: producto, UI, demo y pitch.
- Persona B: analisis tecnico, prompts, backend y reporte de uso de Bob.

Si una persona es mas fuerte en frontend o backend, pueden intercambiar roles, pero manteniendo la separacion de responsabilidades.
